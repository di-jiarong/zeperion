examples/live-version-2/
=========================

Second live end-to-end test of zeperion against itself, run AFTER
Phase 3 fixes (PRs #3, #4, #5) were merged into dev. Compared with
the first live test (examples/live-version-feature/), this run
verifies that the four real bugs that surfaced last time are
actually fixed.


Task this time
--------------

Add a ``--no-pr-pipeline`` boolean flag to ``zeperion run`` so the
multi_agent loop does NOT auto-enter the PR Pipeline subgraph even
when ``GITHUB_TOKEN`` / ``config.github_repo`` are set. This is a
meta-fix: the very bug I myself hit while writing PR #5's tests
(my fixture had ``GITHUB_TOKEN`` exported in the env, multi_agent
auto-ran PR Pipeline, and it pushed 4 spurious "feat: t1" commits
to the remote).

The Developer (claude_code) wrote 6 file changes in 5 minutes:

    zeperion/cli.py                +14 lines  (typer option + forward to detach + pass to graph)
    zeperion/graphs/multi_agent.py  +6 lines  (disable_pr_pipeline kwarg + early return in route_after_tester)
    tests/test_cli_smoke.py         +8 lines  (TestNoPRPipelineFlag::test_run_help_includes_...)
    tests/test_integration.py      +56 lines  (TestNoPRPipeline::test_disable_pr_pipeline_skips_pr_subgraph)
    requirement.txt                +53 lines  (input — moved here from /workspace root)
    .zeperion/config.yaml          +54 lines  (input — moved here from /workspace/.zeperion/)

Code quality is on par with what a careful human reviewer would
produce: the early-return in ``route_after_tester`` is correctly
placed BEFORE the ``config.github_repo or config.github_token``
check; the ``--no-pr-pipeline`` flag is properly forwarded through
``_spawn_detached_run`` so detached runs don't bypass it; and the
integration test sets ``github_token="dummy"`` to force the
GitHub-configured branch and proves the flag dominates.

Limitations not fixed
---------------------

* PR creation itself still failed: ``gh pr create`` returns
  HTTP 401 'Bad credentials'. This is the same Cursor cloud-agent
  token limitation from live test #1, NOT a zeperion bug. Outside
  Cursor's VM (i.e. on a developer laptop with ``gh auth login``)
  this would have completed.
  This PR was therefore opened via Cursor's ManagePullRequest
  tool, not via zeperion's PR Pipeline.

* The auto-PR-pipeline path STILL ran even though I'm the same
  cloud agent who hit this exact footgun while writing PR #5.
  That's because the very task this run is trying to add
  (``--no-pr-pipeline``) doesn't exist yet at the moment of THIS
  run — the Phase 3 fix had to land first before it could be
  used. Tautological. Once this PR is merged, future live tests
  can pass ``--no-pr-pipeline`` to multi_agent and skip the
  subgraph entirely.

How to reproduce
----------------

    # Set env vars (deepseek mapping or real Anthropic key)
    export ANTHROPIC_API_KEY=...
    export ANTHROPIC_AUTH_TOKEN=...
    export ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic   # if using deepseek
    export ANTHROPIC_MODEL=deepseek-v4-pro[1m]                      # ditto
    npm i -g @anthropic-ai/claude-code

    # Set up the live test
    git checkout dev
    cp examples/live-version-2/requirement.txt   ./requirement.txt
    cp examples/live-version-2/config.yaml       ./.zeperion/config.yaml
    rm -rf .zeperion/state

    # Run
    zeperion run --mode multi_agent --thread-id mytest
    zeperion run --mode pr_pipeline --thread-id mytest-pr   # auto-detects sibling

The agents are non-deterministic, so your transcript will not
byte-match this one. But Tester verdicts and commit subjects
should follow the same patterns described above.

AI 长任务无人值守模板（基于文件驱动 + 循环执行）

你会得到两种模式：
1) run_ralph_loop.sh：极简单智能体循环（Ralph 思路）
2) run_multi_agent_loop.sh：主调度 + 计划/开发/测试三子智能体

----------------------------------------
目录结构
----------------------------------------
ai_longrun_harness/
  usage.txt
  requirement_template.txt
  config.env.example
  run_ralph_loop.sh
  run_multi_agent_loop.sh
  prompts/
    master_scheduler_prompt.txt
    planner_prompt.txt
    developer_prompt.txt
    tester_prompt.txt
  state/
    progress.json
    lessons_learned.txt
    task_queue.txt

----------------------------------------
快速开始
----------------------------------------
1) 复制配置
   cp config.env.example config.env

2) 填写需求
   编辑 requirement_template.txt，写清目标、范围、验收标准、约束

3) 准备任务队列（单智能体模式）
   编辑 state/task_queue.txt，每行一个可执行子任务

4) 启动单智能体循环
   bash run_ralph_loop.sh

5) 启动多智能体循环
   bash run_multi_agent_loop.sh

6) 启动 PR 交付管线（开发完成后）
   bash run_pr_pipeline.sh

----------------------------------------
三种模式
----------------------------------------
1) run_ralph_loop.sh：极简单智能体循环（Ralph 思路）
2) run_multi_agent_loop.sh：主调度 + 计划/开发/测试三子智能体
3) run_pr_pipeline.sh：开发完成后的 PR 交付管线
   - 自动 commit + push
   - 创建 PR 到目标分支
   - 轮询 Codex Cloud 👍 审查（最长 30 分钟）
   - 💚 通过 → 启用 GitHub auto-merge
   - ❌ 拒绝/超时 → 保存审查意见到 state/codex_comments.txt → 开发修复后重跑

----------------------------------------
如何接入你的 LLM CLI
----------------------------------------
这套模板不强绑定某个命令行工具，通过 config.env 的命令模板接入：

- SINGLE_AGENT_CALL_TEMPLATE
- MULTI_AGENT_CALL_TEMPLATE

模板中的变量可用：
- MODEL
- PROMPT_FILE
- OUTPUT_FILE
- LOG_FILE
- ROLE
- RESUME_ID_FILE
- REQUIREMENT_FILE

如果你的命令支持 resume，会在 MULTI_AGENT_CALL_TEMPLATE 里读取 RESUME_ID_FILE。

----------------------------------------
关键规则（已固化在脚本里）
----------------------------------------
1) 大任务拆成小任务，循环推进
2) 全部通过文件传递信息，减少上下文占用
3) 测试不过必须回到原开发修复，再由原测试复验
4) 修复次数有限制（默认 3 次），防止死循环
5) 持续写入 lessons_learned.txt，避免重复踩坑
6) 支持不同角色配置不同模型（速度与质量平衡）
7) 角色调用失败自动重试（默认 2 次）
8) 异常会写入 state/workflow_state.json 与 state/errors.log
9) 开发/测试会话 owner 会记录在 workflow_state，FAIL 时继续原会话修复与复测

----------------------------------------
推荐执行习惯
----------------------------------------
1) 先跑多智能体模式，拿到稳定过程
2) 再针对具体项目改 prompt
3) 先把修复上限设小（如 2 或 3），避免失控
4) 每次完成后清理 state/ 下临时输出，保留 lessons 与最终结果

----------------------------------------
中途报错如何处理
----------------------------------------
1) 某个角色调用失败：
   - 自动重试 ROLE_CALL_MAX_RETRIES 次
   - 仍失败则写入 state/errors.log
   - 默认直接停止（CONTINUE_ON_ROLE_ERROR=false）

2) 脚本异常退出：
   - 会触发 trap，进度写为 crashed
   - 详细状态写到 state/workflow_state.json

3) 测试失败：
   - 回到原 developer 会话修复（同一 RESUME_ID_FILE）
   - 再由原 tester 会话复测
   - 超过 MAX_FIX_ATTEMPTS 后停止，防止无限循环

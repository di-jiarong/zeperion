# Lessons Learned - AI Share Platform

## Initial Setup
- Task: Develop HTTP server for AI experience sharing platform
- Tech Stack: Golang + Gin + SQLite
- Started: 2026-05-12

---

## Phase 1 - 项目基础设施搭建 (2026-05-12)

### Developer 经验
1. **单元测试与文件系统解耦**
   - 测试应避免文件系统依赖，使用内存数据库和内联 SQL
   - 创建辅助函数（如 setupTestDB()）提高测试独立性和速度

2. **Makefile 最佳实践**
   - 在开头定义变量（GO := /path/to/go），使用 $(GO) 引用
   - 提高可维护性和跨环境兼容性

3. **代码格式化**
   - go fmt 应在开发过程中持续运行，避免最后阶段才发现问题
   - 建议配置编辑器保存时自动格式化或 git pre-commit hook

4. **修复策略**
   - 严格按照 Tester 的 FIX_REQUEST 执行，不扩大范围
   - 每个修复后立即验证，最后进行综合验证

### Tester 经验
1. **复验流程的重要性**
   - 不能仅依赖 Developer 的自我验证报告
   - 必须实际运行验证命令，确保问题真正修复

2. **测试设计最佳实践**
   - 使用内存数据库 (:memory:) 进行单元测试
   - 测试用例应覆盖核心功能（初始化、健康检查、schema）
   - 测试独立性好，每个测试都有独立的数据库实例

3. **Phase 完成标准**
   - 代码格式正确（go fmt 无输出）
   - 构建成功（make build 生成可执行文件）
   - 测试通过（go test 全部 PASS）
   - 测试覆盖率 > 0%
   - 这四个标准应作为所有 Phase 的基线要求

### 技术决策
- SQLite 连接池配置：MaxOpenConns=25, MaxIdleConns=5
- 数据库文件位置：./data/app.db
- 使用结构化日志（logrus with JSON formatter）
- 索引策略：category, created_at DESC, severity, software, upload_time

## Phase 2 - 后端 API 开发 (2026-05-12)

### Developer 经验
1. **统一响应格式的重要性**
   - 实现 RespondWithError 和 RespondWithSuccess 辅助函数
   - 所有 API 返回统一格式：{code, message, data}
   - 提高前端集成便利性

2. **文件上传安全最佳实践**
   - 文件大小限制防止 DoS 攻击
   - 文件类型白名单防止恶意文件上传
   - 文件名清理防止路径遍历攻击
   - 使用 UUID 重命名文件避免冲突
   - 数据库操作失败时清理已上传文件（事务性）

3. **仓储模式的优势**
   - 将数据库操作封装在 Repository 中
   - Handler 层只关注 HTTP 请求/响应
   - 便于单元测试（可以 mock Repository）
   - 代码结构清晰，职责分离

4. **中间件设计模式**
   - 使用 Gin 中间件实现横切关注点（日志、错误处理、CORS）
   - 中间件顺序很重要：CORS → Logger → ErrorHandler → Routes
   - 统一的日志格式便于后续分析和调试

### Tester 经验
1. **API 集成测试的价值**
   - 单元测试通过不代表 API 能正常工作
   - 必须启动服务器进行端到端测试
   - test_api.sh 脚本覆盖了所有 16 个 API 端点

2. **边界测试的重要性**
   - 测试必填字段验证（空字符串）
   - 测试枚举值验证（无效 severity）
   - 测试资源不存在场景（404）
   - 测试文件大小限制（11MB 文件）
   - 测试文件类型限制（.exe 文件）

3. **功能测试的完整性**
   - 测试分页功能（page, limit）
   - 测试过滤功能（category, severity, software）
   - 测试搜索功能（search 参数）
   - 测试排序功能（sort 参数）
   - 测试下载计数功能（自动更新）

### 技术决策
- API 设计：RESTful，统一 JSON 响应格式
- 文件存储：./data/uploads/config-files/
- 文件大小限制：10MB
- 文件类型白名单：.json, .yaml, .yml, .toml, .txt, .md, .config, .conf, .ini, .xml
- 分页默认值：page=1, limit=20, max_limit=100
- 测试覆盖率：models 52.5%, utils 34.0%, db 15.4%

### 给 Phase 3 的建议
- API 响应格式统一，前端可以统一处理
- 所有列表 API 都返回 {total, items} 格式，便于分页
- 文件上传使用 multipart/form-data，前端需要使用 FormData
- 下载文件时 Content-Disposition 已设置，浏览器会自动下载

## Phase 3 - 前端开发 (2026-05-12)

### Developer 经验
1. **前端架构设计**
   - 单页应用（SPA）使用标签页切换，避免页面刷新
   - 模态框复用，减少 DOM 元素数量
   - 统一的 API 响应格式简化前端处理逻辑

2. **Axios 最佳实践**
   - 使用拦截器统一处理 loading 状态和错误
   - 全局配置 baseURL，避免每个请求重复写路径
   - 文件上传使用 FormData，设置正确的 Content-Type

3. **用户体验优化**
   - 所有操作都有即时反馈（loading、成功、错误提示）
   - 删除操作前弹出确认对话框，防止误操作
   - 文件上传显示进度条，提升用户体验
   - 拖拽上传提供视觉反馈（边框颜色变化、背景色变化）

4. **响应式设计**
   - 使用 Bootstrap 5 栅格系统，自动适配不同屏幕
   - 移动端导航栏折叠为汉堡菜单
   - 使用 @media 查询微调样式

5. **安全性**
   - 使用 escapeHtml 函数防止 XSS 攻击
   - 所有用户输入都进行前端验证（必填字段检查）

### Tester 经验
1. **前端功能验证的完整性**
   - 不能只看代码，必须实际启动服务器并测试所有功能
   - API 集成测试是验证前后端契约的关键
   - 边界测试（文件大小限制、文件类型限制、必填字段验证）非常重要

2. **前端代码质量标准**
   - XSS 防护是必须的，所有用户输入都要经过 escapeHtml 处理
   - 错误处理要完善，所有 API 调用都要有 try-catch
   - 用户体验要好，所有操作都要有即时反馈
   - 响应式设计要完善，支持不同屏幕尺寸

3. **测试策略**
   - 先测试 API 端点（CRUD、分页、过滤、搜索）
   - 再测试边界场景（文件大小限制、文件类型限制、必填字段验证）
   - 最后测试用户体验（loading、提示、确认对话框）
   - 检查浏览器控制台是否有 JS 错误

### 技术决策
- 前端框架：Bootstrap 5 + Axios
- 文件结构：index.html (384行), app.js (784行), style.css (464行)
- 响应式断点：768px, 576px
- CDN 加载：Bootstrap、Axios（加快首次加载）
- 动画效果：fadeIn、hover、transform

### 交付成果
- 3 个主要功能模块（经验点、踩坑点、配置文件）
- 5 个模态框（创建/编辑表单、详情查看、文件上传）
- 完整的 CRUD 操作
- 分页、搜索、过滤功能
- 文件拖拽上传和进度条

## User Auth Feature (2026-05-12)

### Developer 经验
1. bcrypt 已作为 golang.org/x/crypto 的间接依赖存在，直接 import 即可
2. JWT 需手动 go get github.com/golang-jwt/jwt/v5
3. 密码验证统一错误消息防用户枚举是关键安全措施
4. Root 初始化应在 db.InitDB 之后、路由注册前执行
5. db.go 中表数量验证需同步更新（3→4），test_helper.go 同步更新

### Tester 经验
1. 验证密码加密需直接读取 SQLite 数据库确认 $2a$10$ 前缀
2. 防枚举验证需分别测试「不存在用户」和「错误密码」两种场景
3. json:"-" tag 可防止密码哈希泄露到 API 响应
4. 所有现有测试（experiences, pitfalls, config-files）需确认未受影响

### 技术决策
- bcrypt cost = 10 (DefaultCost)
- JWT 24h 有效期，HS256 签名
- JWT_SECRET 环境变量读取，默认值仅开发用
- root 密码 123456 通过环境变量 ROOT_PASSWORD 可覆盖

## Codex 审查节奏教训 (2026-05-12)

### 问题
修复完 push 后立刻 @codex review → Codex 还在审上一轮 → 新的修复提交
被当作新一轮审查 → 产生额外 review 轮次 → 反复循环

### 正确做法
1. push 修复代码 + @codex review 之后，至少等 1 小时再操作
2. Cron 静默轮询，不要手动干预
3. 等 Codex 给出完整结果后再一次性修复
4. 不要在 Codex 审查期间推送新提交

### 关键原则
- 一次只推一次，等 Codex 完整出结果
- 不要因为等不及就手动重新触发 @codex review
- Cron 就是用来等结果的，不要绕过它

## GitHub API 分页截断问题 (2026-05-13)

### 问题
`gh api repos/X/pulls/N/comments` 默认只返回 30 条/页。当评论数超过 30 条时，
新评论被截断，导致轮询始终报告"0 条新评论"，开发者认为没有问题但实际有。
这个问题在 Codex 审查多轮后（累计 >30 条评论）才会暴露。

### 根因
所有 `gh api ...comments` 调用未加 `?per_page=100` 参数。

### 正确做法
- **永远**使用 `gh api "...?per_page=100"` 获取评论
- **永远**用评论总数变化判断是否有新评论，不要用时间过滤
- Cron 轮询时记录上次评论总数，对比当前总数

### Cron prompt 模板必须包含 per_page=100
```
gh api "repos/X/pulls/N/comments?per_page=100" --jq 'length'
```
